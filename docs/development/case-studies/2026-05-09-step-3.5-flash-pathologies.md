# 2026-05-09 — step-3.5-flash multi-turn 失效模式 + 模型层 bug 不应在 nexau 兜底

**TL;DR**: 王晓星接的 step-3.5-flash 在 agent_creator 生产 trace
(`42bce5e2baaf6eaf1528346392aa6062`) 上 100% 必现崩溃。深挖发现叠加三个独立
bug：(a) 我们的 aggregator 漏识别 step 用的裸 `reasoning` 字段；(b) 我们的
`build()` 把 reasoning-only 流当硬错抛 `RuntimeError`；(c) **step-3.5-flash
模型本身**在 3+ 轮 prior tool turn 后会确定性退化成在 `reasoning` 里写 XML
风格的 tool call，而不是用 OpenAI 标准 `tool_calls`。Bisect 出阈值是 8 条消
息，所有 prompt-engineering workaround（anti-XML system prompt、
`tool_choice=required`、`tool_choice={specific}`、`temperature=0`）全部失效。
PR #542 修了 (a)+(b)；(c) 不在 nexau 修，反馈给 step 团队。

**Date**: 2026-05-09
**Driver**: 生产 trace 中 agent_creator 整 trace 死掉，王晓星私聊报告。
**Files**: PR #542 ─ aggregator/model_response 修复 + 6 个 step35 SSE
fixture（含 production XML-in-reasoning 复现）。

## 触发症状

Langfuse trace `42bce5e2baaf6eaf1528346392aa6062`：

- 37 次 LLM call
- 32 次"成功"，但**0 次用了标准 `tool_calls` 字段**（全是 text content）
- 5 次 ERROR：`Choice 0 was never aggregated with any content`
- agent_creator 死亡

## 真因分解（三层）

### bug (a) ─ 漏识别 `reasoning` 字段（nexau 这边）

`OpenAIChatCompletionAggregator._extract_reasoning_delta` docstring 写"
`reasoning` flat — all routed through `_extract_reasoning_delta`"，**但代
码 line 439 实际只看 `reasoning_content` 和 `reasoning_details`，没看裸
`reasoning`**。docstring 撒谎。Step 用裸 `reasoning`，所以 chain-of-thought
全丢，前端 trace 也没东西显示。

### bug (b) ─ reasoning-only stream 抛硬错（nexau 这边）

`_ChoiceAggregator.build()` 检查"content is None and refusal is None and
no tool_calls" → 抛 `RuntimeError`。但 thinking model 在
`finish_reason=length` 截断、或 reasoning-only finish=stop（见 bug c）下
都是合法的 vendor 输出 —— 我们把唯一可用的 reasoning 文本扔了，UI 啥都
看不到。

### bug (c) ─ step-3.5-flash 模型层退化（不在我们这）

**100% 复现条件**（5/5 trials at temperature=0.2）：
对话历史中有 **3+ 个 prior assistant tool turn** 时，模型不再用 OpenAI 标
准 `tool_calls` 字段，而是把工具调用以 XML 形式写进 `reasoning`：

```xml
<function=run_shell_command>
<parameter=command>find /tmp -type d</parameter>
<parameter=description>list dirs</parameter>
</function>
</tool_call>
```

伴随：

- `finish_reason=stop`
- `delta.content=""`（不是 None）
- `delta.tool_calls=[]`
- `usage.completion_tokens_details.reasoning_tokens=0`（vendor 自己分类
  错，把 reasoning token 算成 completion token）

**Bisect 出阈值**：用 production trace 切片测：

| 切片 | 历史 role 模式 | 结果 |
|---|---|---|
| sys+user only (2 msgs) | s\|u | ✅ 标准 `tool_calls` |
| +last_3 (5 msgs) | s\|u\|t\|a\|t | ✅ 标准 `tool_calls` |
| +last_4 (6 msgs) | s\|u\|a\|t\|a\|t | ✅ 标准 `tool_calls` |
| +last_5 (7 msgs) | s\|u\|t\|a\|t\|a\|t | ✅ 标准 `tool_calls` |
| **+last_6 (8 msgs)** | s\|u\|a\|t\|a\|t\|a\|t（3 prior asst tool 轮）| ❌ XML in reasoning |
| +last_8 ~ +last_73 | longer | ❌ XML in reasoning |

**Prompt-engineering 全部失效**：

| Workaround 尝试 | 结果 |
|---|---|
| 显式 system prompt 禁止 XML / 强调用 OpenAI tool_calls | ❌ 仍 XML |
| `tool_choice="required"` | ❌ 仍 XML |
| `tool_choice={"type":"function","function":{"name":"list_directory"}}` 强制特定 tool | ❌ 仍 XML |
| `temperature=0` | ❌ 仍 XML |

这些都验证了 bug (c) 是模型 weights 层面的硬退化，不是 sampling 偶然或 prompt 引导问题。

## 决策

PR #542 修 (a) + (b) + 加 **executor 层 vendor-pathology guard**，**不写
XML parser 兜底 (c)**。原因：

1. **契约层不该容忍模型层 bug** —— 我们对 vendor 的承诺是"按 OpenAI 协议
   返回 tool_calls 我们就处理"。一旦写 XML 兜底，反过来诱导 vendor 不修。
2. **XML 形式不稳定** —— 跨模型、跨版本可能变；维护成本高。
3. **bug (c) 的根因是模型不可用于 multi-turn agent driver**，光修 nexau 这
   层即使把 XML parse 出来也无法掩盖：模型在长 context 下表现还会出现别的
   pathology。

### Aggregator-level placeholder substitution（同 PR）

光让 aggregator 不抛 `RuntimeError` 还不够 —— `ModelResponse` 带
`content=None` + `tool_calls=[]` 的"成功"返回会进 history 然后炸下一轮：

| Vendor | 收到 history 里有 `assistant content=""` / `[]` |
|---|---|
| step-3.5 / OpenAI compat | ✅ 接受，但下一轮 LLM 大概率又是空（recursion）|
| **Anthropic** | ❌ HTTP 400（messages.X.content 不能空）|
| **Gemini** | ❌ HTTP 400（parts 不能空，schema 严格）|

且 `LLMFailoverMiddleware` 触发不到（不是 5xx，是"成功"返回），同 session
跨 turn 续接、跨 vendor failover、replay 都是雷。

**架构选择**：在 `_ChoiceAggregator.build()` 内部直接补 `[empty]` 占位 ——
"空 stream 构造出可用 message" 是 aggregator 的合理职责，不通过 middleware。

> ⚠️ 试错过程：最早实现是 `EmptyPayloadGuardMiddleware`（默认 builtin 注入到
> `wrap_model_call` hook），结果触发了 LLM caller 的一个隐藏 fork ——
> `if self.middleware_manager:` 真假分支决定走 sync 桥接 vs 原生 async
> ([llm_caller.py:892](../../nexau/archs/main_sub/execution/llm_caller.py)).
> 默认注入让所有 agent 都从 async 路径切到 sync 桥接，副作用：(a) 测试 spy 装在
> async client 上不再触发；(b) cancellation 信号通过 thread pool 传播延迟，
> `test_graceful_stop` 等 e2e timeout 失败。教训：**在 nexau 当前架构下，默认
> 注入任何 middleware 都会改变所有 agent 的 LLM 调用路径**。把 vendor-pathology
> 检测放在 aggregator 内部就避开了这个 quirk —— aggregator 跑在 LLM call 内部，
> 不通过 middleware 机制。

```python
# nexau/archs/llm/llm_aggregators/openai_chat_completion/openai_chat_completion_aggregator.py
_EMPTY_CONTENT_PLACEHOLDER = "[empty]"  # 7 chars, ~3 BPE tokens

def build(self) -> ChatCompletionChoice:
    has_any_signal = (content / refusal / tool_calls / reasoning_content / reasoning_details)
    if not has_any_signal:
        raise RuntimeError(...)  # 真·空 stream（连 reasoning 都没）→ fail loud
    if (
        not self._value.message.content
        and self._value.message.refusal is None
        and not self._tool_call_aggregators
    ):
        self._value.message.content = _EMPTY_CONTENT_PLACEHOLDER
    ...
```

设计要点：

- **短 ASCII placeholder `[empty]`** —— BPE 约 3 token；session resume 续接
  prompt-cache miss 增量极小。详见下面 §"prefix cache 影响"。
- **`[...]` 包裹** —— UI 一眼可识别，grep 友好。
- **诊断细节不在 wire 上** —— reasoning 文本仍然附在
  `message.reasoning_content` 字段（OpenAI/Step serializer 白名单 drop，
  但 Anthropic serializer 会作为 ThinkingBlock 保留 + Langfuse trace 完整捕获）。
- **只 fix `openai_chat_completion`** —— 其他 provider（Anthropic/Gemini/
  OpenAI Responses）的 aggregator 暂未发现同样问题；将来如果遇到再各自
  patch（同样位置：自己的 build()）。

### 原始返回的审计

| Sink | 能否恢复原始 reasoning / raw payload |
|---|---|
| **Langfuse trace** | ✅ 完整保留 —— Langfuse 在 LLM caller 层独立捕获，发生在 aggregator 之前 |
| **`message.reasoning_content`**（in-memory + DB persisted as ReasoningBlock）| ✅ 保留原始 reasoning 文本 |
| **`message.content`**（wire payload to next-turn LLM）| ❌ 只剩 `[empty]` placeholder（这是设计目的）|

**实际意义**：开 Langfuse → 完全可审计；不开 → 仍可从 ReasoningBlock 在
DB 里读到原始 reasoning（Anthropic 序列化时还会作为 ThinkingBlock 回送，
OpenAI/Step 不会 — 由 serializer 白名单决定）。

下一轮（session resume）的 cache 命中位置和"不加 guard / 不 append /
让空消息进 history"三种方案完全相同 —— 都断在那个失败的 turn 处：

```
[sys, user, ...prior_real_turns, FAILED_TURN_PLACEHOLDER, new_user]
                                  ↑
                         cache 在这里断
```

prefix `[sys, user, ...prior_real_turns]` 仍然 100% 命中（生产 trace 里这部
分约 49K tokens / 99% cache hit）。`[empty]` 占位带来的 cache miss 只是
3 个 token 量级，可忽略不计。

## 用法建议（直到 step 团队修 (c)）

| 用法 | 推荐？ | 原因 |
|---|---|---|
| 单轮 chat / 文本生成 / reasoning helper | ✅ | bug (c) 不触发 |
| 短 context（< 5K tokens）+ 单 tool 的简单调用 | ✅ | bug (c) 不触发 |
| Multi-turn agent driver（agent_creator / DevBox / code agent 等）| ❌ | bug (c) 必现，agent 跑不动 |
| Code-editing / file-manipulation agent | ❌ | 必然 long context + multi-turn tool |

## 教训 / 反模式

1. **Aggregator docstring 撒谎不会被 CI 抓** —— bug (a) 的 docstring 在
   "reasoning flat" 上的承诺写了一年没人发现。修复后续：在 fixture 里
   覆盖每个声称支持的 wire shape（recordings/step35_*.sse 现在覆盖了
   bare `reasoning`）。
2. **"Choice never aggregated" 这种"硬错"在 thinking model 时代是错的设
   计** —— thinking model 可以合法地只产出 reasoning 不产出 content
   （max_tokens 太小、网络抖动、模型当机内 bug）。把它视作 fatal 等于
   把可用的 chain-of-thought 信号全扔了。所有"choice empty"类的早期 raise
   都该重审一次。
3. **Bisect 是诊断 LLM 行为退化的首选方法**：从最小切片
   (`sys+user`) 开始倍增 history，找到阈值。比"试 prompt engineering"
   先做。
4. **prompt-engineering workaround 不是排除模型 bug 的证据** —— 但
   workaround 全部失效**是**模型 bug 的证据。下次遇到模型怪异行为，
   先 bisect 找触发条件，再用 4 种 workaround 排除（system prompt /
   tool_choice / temperature / api param），再下结论。
5. **XML-in-reasoning 这种 vendor 行为不应该在 nexau 兜底** —— 默认契
   约是 OpenAI 协议；vendor 不守契约就让它失败、报回 vendor、不要在客
   户端写 parser 救场。一旦写救场代码，永远撕不掉。
